# RFC NNNN: Title

- **RFC number**: NNNN
- **Start date**: YYYY-MM-DD
- **Target spec version**: 0.x

## Summary

One paragraph explaining the proposal. What is being changed, added, or removed, and why.

## Motivation

Why is this change needed? What problem does it solve? What use cases does it enable or improve? What is the impact of not doing it?

## Detailed design

The normative description of the change. This is the meat of the RFC. Be specific:

- For protocol changes: describe the new lifecycle step, role obligation, or failure mode.
- For schema changes: list new/modified fields with type, required/optional, default value, and constraints.
- For adapter changes: describe new behavior, exit codes, or env vars.
- For instrumentation changes: describe new event types or modified fields.
- For conformance changes: describe the new level requirement or modified vector format.

Include examples where they clarify intent.

## Drawbacks

What are the costs of this change? Complexity, backward compatibility, implementation burden, potential for misuse?

## Alternatives

What other approaches were considered and why were they rejected? What is the impact of not doing this RFC at all?

## Unresolved questions

What aspects of the design are still unclear or contentious? What needs to be decided before this RFC can be accepted? What can be deferred to a follow-up RFC?
